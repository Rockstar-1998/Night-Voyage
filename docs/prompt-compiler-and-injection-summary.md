## 一、Prompt Compiler 是什么

Prompt Compiler 不是 system prompt 本身。

它是：

**在真正请求模型前，把角色卡、世界书、预设、历史、向量检索结果、当前输入等来源的数据统一收集、排序、裁剪、编译成最终 `messages[] + params` 请求的后端总装配器。**

在当前代码里，最早期的雏形位于：
- [`compile_chat_messages()`](src-tauri/src/services/prompt_compiler.rs:8)

Prompt Compiler 最终应负责：

1. 收集输入源
2. 宏展开
3. 世界书 / 正则规则匹配
4. 历史层分层编译
5. token 预算裁剪
6. provider 能力矩阵判断
7. prefill 编译
8. 输出结构化请求体

## 二、历史层最佳分层

### 第 0 层：最近原文窗口
- 永远保留最近 N 轮原始对话
- 不压缩
- 保证当前轮连续性
- 注入层：`assistant/user history`

### 第 1 层：手动总结层（可选）
- 用户愿意时可以自己写
- 高权威
- 绝不能强制用户使用
- 一旦存在，优先于 AI 总结
- 注入层：`system`

### 第 2 层：AI 总结层
建议拆成两类：

#### 2.1 AI 剧情总结
- 总结区间剧情主线
- 注入层：`system`

#### 2.2 AI 角色状态总结
- 总结角色因剧情、关系、创伤、目标变化而发生的可变人格变化
- 这层允许覆盖角色卡中的可变部分
- 注入层：`system`

### 第 3 层：向量细节检索层
- 不是“总结器”，而是“细节召回层”
- 只负责从大量历史中找和当前最相关的细节片段
- 不负责定义主线或人格
- 注入层：`system` 中的低权重参考区块

## 三、角色卡是否允许被总结覆盖

结论：

- **允许覆盖角色卡中的可变部分**
- **不建议覆盖角色卡中的基础硬设定**

### 角色卡中的硬设定
通常不应被普通总结轻易覆盖：
- 名字
- 身份
- 出身
- 物种 / 职业 / 能力基础
- 世界观定位

### 角色卡中的软设定
应当允许被剧情发展覆盖：
- 性格倾向
- 说话口吻
- 信任关系
- 当前目标
- 创伤后反应
- 成长后的价值观

### 推荐做法
- 角色卡 = 基础人格底座
- 世界书 = 世界初始状态
- 世界变量层 = 记录相对于初始状态的变化

不要直接把角色卡或世界书原文改掉，而应在 Prompt Compiler 中叠加"世界变量层"来记录变化。

## 四、向量化到底是什么

向量化不是自动总结。

它更准确的说法是：
- 语义编码
- 相似度检索
- 细节召回

工作流程：
1. 把历史切成片段
2. 用 embedding 模型转成向量
3. 存入检索层
4. 当前轮输入也转成查询向量
5. 比较相似度
6. 取最相关的几条片段
7. 做重排 / 去重 / 裁剪后补进 prompt

它擅长：
- 找细节
- 找相似语义片段

它不擅长：
- 稳定地主导剧情主线总结
- 定义角色当前最终人格

所以 Night Voyage 里，向量层只应做：
- 细节补丁层
- 记忆碎片检索层

不能让它越权定义主线和人格，否则容易 OOC。

## 五、和 SillyTavern 的区别

### SillyTavern
更像：
- Prompt 实验台
- 大量灵活手工拼接
- 高自由
- 适合 prompt 工程实验

优势：
- 自由度高
- 改 prompt 很快
- 生态插件丰富

弱点：
- 很容易把所有信息拼成大杂烩
- 长历史下不易做稳定层级控制
- 更依赖人工约定注入顺序与权重

### Night Voyage
更像：
- 后端主导的结构化 Prompt Compiler
- 有 authority 层级的叙事编译器
- 支持长历史、大世界书、大角色卡
- 支持角色成长与多人会话

优势：
- 结构清晰
- 可调试
- 更适合做 token 预算
- 更适合长期扩展 agent / retrieval / summaries

一句话：
- SillyTavern 更像 prompt 工作台
- Night Voyage 更像结构化叙事编译器

## 六、每一层最推荐注入到哪一提示词层

### 1. 预设规则层
- 内容：风格、格式、输出约束、节奏要求
- 最推荐注入层：`system`

### 2. 角色卡基础层
- 内容：角色是谁、基础人格底座、初始身份
- 最推荐注入层：`system`

### 3. 世界书命中层
- 内容：当前轮触发的设定条目
- 最推荐注入层：`system`
- 顺序：角色卡基础层之后、世界变量层之前

### 4. 世界变量层
- 内容：记录世界中地点、人物等变量因剧情推进和时间流逝而发生的相对于原角色卡和世界书的变化
- 示例：某个地点从"荒废"变为"繁华"、某个NPC从"陌生人"变为"挚友"、某个时间节点后的季节/天气变化
- 最推荐注入层：`system`
- 顺序：世界书命中层之后

### 5. 剧情总结层
- 内容：压缩后的剧情主线摘要
- 最推荐注入层：`system`

### 6. 向量细节层
- 内容：检索出来的少量高相关细节片段
- 最推荐注入层：`system` 中的低权重参考区
- 数量必须严格限制

### 7. 预设 few-shot 示例层
- 内容：示例 user / assistant 对话
- 最推荐注入层：`assistant/user examples`
- 不推荐塞进 `system` 文本里

### 8. 最近原文窗口
- 内容：最近 N 轮原始对话
- 最推荐注入层：`assistant/user history`

### 9. 当前轮输入
- 内容：这次真正要回答的输入
- 最推荐注入层：最后一条 `user`

## 七、预设系统到底够不够灵活

结论：

**可以足够灵活，但不能做成毫无结构的自由文本大框。**

最好的形态是：

- 底层编译协议固定
- 上层模块自由组合

### 固定的是
- 如何保存
- 如何排序
- 如何编译
- 如何裁剪
- 如何被 provider override 覆盖

### 灵活的是
- block 数量
- block 类型组合
- few-shot 数量
- 启用顺序
- 是否继承父预设
- 是否有 provider-specific override

一句话：

**预设不是死模板，而是一组可排序、可开关、可继承、可覆盖、可编译的生成策略模块。**

## 八、预填充、正则、宏的最佳定位

### 1. Prefill
本质：
- 输出起始控制器

作用：
- 给模型一个起始前缀
- 要求模型从这个片段后继续生成

最适合归属：
- 预设系统
- 或当前轮特殊生成控制

最推荐注入层：
- **不是提示词层**
- 首选：provider adapter / 请求参数编译层
- 次选：隐藏 assistant seed
- 最后才考虑变成 system 说明

### 2. Regex
本质：
- 规则匹配器 / 校验器 / 轻量后处理器

适合用途：
- 世界书触发规则
- 输出后处理规则
- 结构校验规则

最推荐注入层：
- **不进任何提示词层**
- 它属于编译器执行层

### 3. Macro
本质：
- 模板展开器

适合用途：
- 变量替换
- 条件展开
- 列表展开

推荐引擎：
- [`minijinja`](plans/performance-architecture.md)

最推荐注入层：
- **宏本身不进提示词层**
- 它先在 Prompt Compiler 中展开
- 展开后的文本再进入 `system` / `examples` / `summary` 等层

## 九、三者如何一起工作

推荐流水线：

1. 读取预设 block
2. 做宏展开
3. 跑世界书触发规则（关键词 / 正则）
4. 组装角色卡、角色状态、历史总结、向量细节
5. 做 token 预算裁剪
6. 判断 prefill 能力矩阵
7. 编译成 provider-specific request
8. 发送模型
9. 输出后执行 regex 后处理 / 校验

也就是说：
- 宏先于正则
- 正则先于最终注入
- prefill 晚于正文编译完成
- prefill 作用在输出起点
- regex 作用在规则匹配和清洗
- macro 作用在内容生成

---

## 十、Night Voyage 当前最推荐的完整注入顺序

1. 预设规则层 -> `system`
2. 角色卡基础层 -> `system`
3. 世界书命中层 -> `system`
4. 世界变量层 -> `system`
5. 剧情总结层 -> `system`
6. 向量细节层 -> `system` 中的参考区
7. 预设 few-shot 示例 -> `assistant/user examples`
8. 最近原文窗口 -> `assistant/user history`
9. 当前轮输入 -> 最后一条 `user`

---

## 十一、最终一句话总结

**Night Voyage 的最佳实践不是把所有内容都堆进一个 system prompt，而是通过 Prompt Compiler 把预设、角色卡、角色成长、世界书、剧情总结、向量细节、最近原文和当前输入分层编译，并让宏、正则、预填充分别作为模板层、规则层和生成控制层工作。**
