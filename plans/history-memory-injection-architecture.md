# Night Voyage 历史记忆、向量检索与内容注入架构

## 文档目标

本文用于明确 Night Voyage 在超长历史对话、角色成长、世界书命中、预设注入与向量检索场景下的推荐架构。

本文重点回答 4 个问题：

1. 历史层如何分层管理
2. 向量化到底在做什么
3. 这种架构和 SillyTavern 有什么区别
4. 世界书、角色卡、预设、历史等内容是不是都要注入到 system prompt

---

## 核心结论

Night Voyage 不应该把历史压缩问题简化成“人工总结 / AI 总结 / 向量总结 三选一”。

最佳实践是：

- 最近原文窗口：保留最近 N 轮原文
- 手动总结：可选的高权威总结层，绝不强制用户使用
- AI 总结：默认的自动压缩层，分剧情总结和角色状态总结
- 向量检索：细节召回层，不替代总结主线
- 角色卡：基础角色底座
- 角色状态覆盖层：允许角色因经历、关系和剧情发展而变化
- 世界书：按需命中注入的外部设定层
- 预设：生成策略与风格规则层

也就是说：

- 角色卡定义“这个角色最初是谁”
- 角色状态总结定义“这个角色现在变成了什么样”
- 世界书定义“当前上下文还需要补充哪些设定”
- 预设定义“模型该怎么写”
- 向量检索只负责“把相关细节找出来”，不负责定义主线和人格

## 一、向量化到底是怎么工作的

### 1. 它不是“自动总结”

向量化更准确的说法是：

- 语义编码
- 相似度检索
- 细节召回

它最擅长的是：

- 从大量历史中找到和当前输入语义最接近的片段
- 把这些片段作为补充信息送回模型

它不擅长的是：

- 直接给出稳定、完整、逻辑连续的长程剧情总结
- 定义某个角色在当前阶段的最终人格
- 决定什么才是绝对 canon

所以 Night Voyage 里不要把它定义为“主总结器”，而应定义为：

- 历史细节检索层
- 记忆碎片召回层

### 2. 工作流程

#### 第一步：切片

把历史会话拆成很多小片段，例如：

- 一轮对话
- 一段关键事件
- 一条角色态度变化
- 一段重要描述
- 一个设定事实片段

#### 第二步：嵌入编码

把每个片段送进 embedding 模型，转成一个浮点向量。

这个向量可以理解为“语义坐标”。

#### 第三步：存储

把以下信息一起保存：

- 原始片段文本
- 来源会话 / 轮次 / 消息 ID
- 角色 ID
- 片段类型
- 向量
- 置信度或标签

#### 第四步：查询向量生成

用户发来新一轮输入时，把“当前轮输入 + 当前上下文目标”也转成一个查询向量。

#### 第五步：相似度比较

拿查询向量去比所有历史向量，取最相近的几条片段。

常见相似度度量：

- cosine similarity
- dot product
- euclidean distance

#### 第六步：重排与裁剪

不能把相似度结果直接原样塞给模型。

还要做：

- 按角色权限过滤
- 按时间权重过滤
- 按片段类型过滤
- 按 token 预算裁剪
- 去重

#### 第七步：补充注入

最后把这些片段作为“细节补丁”注入 prompt，而不是代替全部历史。

## 二、为什么向量检索容易导致 OOC

OOC 不是因为向量本身“坏”，而是因为它很容易被错误使用。

### 常见问题

1. 检错角色
- 角色 A 拿到了角色 B 的记忆片段

2. 检错阶段
- 检到的是早期片段，但当前角色已经成长变化

3. 检错语境
- 语义类似，但剧情并不适用

4. 检索层越权
- 把细节召回层当成主线定义层

### 解决原则

- 向量层只能补细节，不能定义人格主线
- 角色状态的最终权威应来自角色卡 + 角色状态总结
- 高优先级记忆要有显式 authority 标记
- 检索前先做角色和时间范围过滤

---

## 三、历史层的最佳分层

推荐分成 4 层。

### 第 0 层：最近原文窗口

作用：

- 保持最近对话连续性
- 保证语气、动作、细节不丢

特点：

- 不总结
- 不压缩
- 永远保留最近 N 轮原文

### 第 1 层：手动总结层（可选）

作用：

- 用户高精度表达真正想保留的剧情与角色变化

特点：

- 高权威
- 绝不强制使用
- 只在用户愿意时才创建
- 一旦存在，应优先于 AI 总结

### 第 2 层：AI 总结层

这里建议再拆成两类：

#### 2.1 剧情总结
描述：

- 这个区间发生了什么
- 世界状态如何推进
- 有哪些重要事实变化

#### 2.2 角色状态总结
描述：

- 这个角色在该区间后性格如何变化
- 对某些人或某些事的态度如何变化
- 当前目标、情绪、信任关系如何变化

这一层非常关键，因为它负责处理你提出的“角色会因经历改变性格”问题。

### 第 3 层：向量细节检索层

作用：

- 把可能被压缩总结遗漏的关键细节捞出来
- 例如一句承诺、一个细节动作、一个物品规则、一个伏笔

它的职责是：

- 补细节
- 不定义主线
- 不直接覆盖高权威设定

## 四、角色卡是否允许被总结覆盖

答案是：

- **允许覆盖角色卡中的可变部分**
- **不建议覆盖角色卡中的基础硬设定**

### 角色卡内容分两类

#### 1. 硬设定
通常不应被普通总结轻易覆盖：

- 名字
- 身份
- 出身
- 物种 / 职业 / 能力基础
- 世界观定位

#### 2. 软设定
应当允许被剧情发展覆盖：

- 性格倾向
- 说话口吻
- 防备程度
- 对其他角色的态度
- 当前目标
- 心理创伤后反应
- 成长后的价值取向

### 推荐做法

不是直接修改角色卡原文，而是：

- 角色卡 = 基础底座
- 角色状态总结 = 覆盖层

这样既能保留角色最初设定，也能让模型知道这个角色已经发生变化。

---

## 五、手动总结的产品原则

这里明确一条硬约束：

- **绝不能把手动总结设计成系统必经步骤**

正确方式应该是：

- 默认由系统自动做 AI 总结与向量检索
- 手动总结只是可选增强层
- 不做“你不总结系统就不好用”的依赖

也就是说：

- 手动总结可以有
- 但必须是 optional
- 不得强迫用户劳动

## 六、推荐的注入顺序

Night Voyage 的内容注入顺序建议是：

1. 预设层
2. 角色卡基础层
3. 角色状态覆盖层
4. 世界书命中层
5. 剧情总结层
6. 向量细节层
7. 最近原文窗口
8. 当前轮输入

### 每一层职责

#### 1. 预设层
描述：

- 写作风格
- 输出约束
- 节奏要求
- 格式规则
- 参数语义归属（如 temperature / max tokens 属于预设层）

#### 2. 角色卡基础层
描述：

- 角色名称
- 基础身份
- 基础人格底座
- 基础口吻

#### 3. 角色状态覆盖层
描述：

- 成长变化
- 创伤变化
- 关系变化
- 阶段性性格偏移

#### 4. 世界书命中层
描述：

- 当前轮触发的设定条目
- 不全量注入，只注入命中项

#### 5. 剧情总结层
描述：

- 历史主线骨架
- 阶段性剧情状态

#### 6. 向量细节层
描述：

- 召回少量高相关细节片段
- 用来补摘要层丢失的精细信息

#### 7. 最近原文窗口
描述：

- 最近 N 轮原始消息
- 保证连续性

#### 8. 当前轮输入
描述：

- 这次真正要回复的内容

## 七、这些内容是不是都要注入到 system prompt？

答案是：**不是。**

### 不能全部都放进 system prompt

如果把所有东西都塞进 system：

- system prompt 会极度膨胀
- 历史消息会失去“对话语义”
- 一些模型对 system 权重过高，反而压掉自然对话感
- 角色成长、摘要、细节会全部变成“规则文本”，不自然

### 推荐的注入通道划分

#### 适合放在 system 的内容

- 预设规则
- 角色卡基础层
- 角色状态覆盖层
- 世界书命中条目
- 剧情总结层中的高层概括
- 安全约束与格式约束

#### 不适合放在 system 的内容

- 最近原文窗口
- 当前轮输入
- 大量历史原始对话
- 所有向量召回片段全量堆叠

这些应继续保留在 `messages[]` 的 user / assistant 历史结构中。

### 向量细节层怎么放

推荐：

- 默认作为“低优先级补充上下文块”注入
- 可以是 system 中一个单独命名的 `Retrieved Details` 区
- 但数量必须非常少
- 不能替代最近原文窗口

### 当前 Night Voyage 最稳妥的做法

在 OpenAI 兼容链路里，初期可采用：

1. system message：
   - preset
   - character base
   - character state overlay
   - matched world book
   - summary block
   - retrieved detail block（少量）
2. assistant / user history：
   - 最近原文窗口
3. current user：
   - 当前轮输入

也就是说：
- 高层规则与压缩上下文适合放 system
- 原始对话历史和当前输入仍然保留在原本角色消息通道里

所以正确答案是：

- **不是所有注入内容都进 system prompt**
- **只有规则性、概括性、背景性内容优先进 system**
- **原始对话历史和当前轮输入必须保留在对话消息层**

## 八、Night Voyage 与 SillyTavern 的区别

### SillyTavern 的典型特点

SillyTavern 更像：

- 高度灵活的 prompt 拼装工作台
- 偏前端与模板驱动
- 用户对注入顺序有很高自由度
- 社区生态强、扩展快、实验性强

它的优势：

1. 灵活
2. 好试验
3. 各类 prompt 格式、jailbreak、宏、插件非常丰富
4. 对重度手调用户很友好

它的弱点：

1. 容易把所有信息拼成一锅大 prompt
2. 大规模长历史时稳定性和结构化控制不一定最好
3. “哪一层信息优先级更高”往往靠人工约定，而不是后端强约束
4. 在多人会话、角色成长、权威总结、向量细节分层等方面，默认模型通常不够严格

### Night Voyage 的目标区别

Night Voyage 更像：

- 后端主导的结构化 Prompt Compiler
- 有明确 authority 层级的剧情引擎
- 会话是角色卡实例
- 支持多人轮次、状态覆盖、世界书按需命中、未来 agent 调度

它的优势：

1. 结构化强
2. 权威层级清晰
3. 更适合超长历史、大世界书、大角色卡
4. 更适合多人会话与角色成长覆盖
5. 更容易做性能优化和跨平台稳定控制

它的代价：

1. 实现复杂度更高
2. 调优速度比 SillyTavern 慢
3. 用户即兴修改 prompt 结构的自由度没有那么野

### 一句话区别

- **SillyTavern 更像 Prompt 实验台**
- **Night Voyage 更像有剧情状态机、记忆层级和后端编译器的结构化叙事引擎**

### 各自更适合什么

#### SillyTavern 更适合
- 快速试 prompt
- 高自由手工调试
- 社区插件驱动玩法
- 个人单机实验

#### Night Voyage 更适合
- 长剧情跑团
- 角色成长与关系演化
- 超长历史与大设定管理
- 多人会话
- 未来 agent 化和向量检索精细分层

## 九、我给 Night Voyage 的最终推荐

### 推荐历史架构

- 最近原文窗口：默认保留
- AI 剧情总结：默认存在
- AI 角色状态总结：默认存在
- 手动总结：可选高权重覆盖层，绝不强制
- 向量层：只做细节检索，不做主线定义

### 推荐注入原则

- 预设决定“怎么写”
- 角色卡决定“最初是谁”
- 角色状态总结决定“现在变成什么样”
- 世界书决定“当前场景还要补什么设定”
- 向量层只补细节，不定义主线
- 最近原文和当前轮输入保留在消息通道，不全部塞进 system

### 最终一句话

**Night Voyage 最佳实践不是把所有内容都堆进一个 system prompt，而是把预设、角色卡、角色成长、世界书、剧情总结、向量细节、最近原文和当前输入分层编译，并通过不同优先级和不同消息通道送进模型。**

这套方式比简单拼 prompt 更稳，也比把向量层当万能总结器更不容易 OOC。
