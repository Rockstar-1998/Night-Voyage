# Night Voyage 预设系统易用性改进原则

## 文档目标

本文用于沉淀 Night Voyage 预设系统在“直观性、易用性、作者自由、结构化架构”之间的平衡原则。

它不直接替代 [`docs/preset-system-architecture.md`](docs/preset-system-architecture.md) 的底层协议设计，而是补充回答下面这些产品问题：

1. 预设应该怎么让普通作者一看就会用
2. 预设应该如何避免 ST 式“靠经验拼接”的理解负担
3. 平台应该替作者做哪些解释，哪些仍交给作者自由发挥
4. 语义组、自由 block、条目锁、互斥组之间应如何分工

---

## 核心结论

Night Voyage 的预设系统不应当以“编程思维”来组织，而应当以：

- 写作意图
- 叙事目标
- 创作效果
- 适用场景

来组织。

一句话说：

**Night Voyage 的预设应该像写导演要求，而不是像写程序。**

这意味着：

- 用户操作的是“效果”与“意图”
- 平台负责把规则讲人话、把冲突显性化、把边界讲清楚
- 作者负责填写文风、玩法、适用场景与风险说明

---

## 已确认的产品方向

## 1. 一级主分组应清晰、强语义、可互斥

以下能力适合做成一级语义组：

- 单人 / 多人
- SFW / NSFW

原因：

- 它们决定的是整套预设的主要使用语境
- 它们不是普通风格补丁，而是“这套预设属于哪一类”的根定义
- 它们最适合做成互斥组，而不是平铺开关

推荐实现：

- `会话类型` 语义组
  - `单人预设`
  - `多人预设`
- `内容类型` 语义组
  - `SFW`
  - `NSFW`

说明：

- 前台展示上应强隔离
- 编译层不必做绝对硬封锁
- 推荐通过标签、筛选和默认推荐路径把它们自然分开

---

## 2. 语义子项应带人话说明，不应只显示名字

每个语义子项至少应具备三类信息：

1. 一句话说明
2. 一个超短示例
3. 一个或多个适合场景标签

例如：

- `第二人称`
  - 说明：把玩家角色称作“你”
  - 示例：你推门进去，灯光正落在桌面上
  - 场景：跑团推荐、代入互动推荐
- `限知第三人称`
  - 说明：以第三人称叙述，但只写角色当前知道的信息
  - 示例：她看见门开了，却还不知道来者是谁
  - 场景：剧情文推荐、悬念感推荐

推荐交互：

- 条目 hover 气泡
- 简短帮助文案
- 适用场景小标签

---

## 3. 平台负责解释，不替作者代写内容

Night Voyage 不应当过度替作者决定什么组合“不好”。

更合理的职责边界是：

### 平台负责

- 解释规则是什么意思
- 告诉用户当前是否生效
- 告诉用户冲突来自哪里
- 标记 provider 兼容性问题
- 可视化 token 占用与复杂度

### 作者负责

- 文风内容本身
- 适用场景
- 风险说明
- 补充玩法说明
- 自由 block 的具体写法

一句话说：

**平台负责把规则讲清楚，作者负责把内容写精彩。**

---

## 4. 语义组与自由 block 的边界

这部分不应用纯工程术语硬塞给作者，推荐采用更直观的话术：

- 安全区
- 自由区

### 安全区

适合放：

- 条目锁条目
- 互斥条目
- 高频基础规则
- 主提示词骨架
- 一级语义组

特征：

- 高优先级
- 易冲突
- 需要较强约束
- 平台应尽量帮用户看懂

### 自由区

适合放：

- 自定义补丁
- 实验条目
- 作者个性化玩法
- 临时增强 block

特征：

- 灵活
- 可试验
- 不必过度模板化
- 平台主要提供编辑容器与解释位

一句话说：

**安全区负责稳定， 자유区负责发挥。**

---

## 5. 条目锁不应只有“禁止”，还应有人性化出口

条目锁已经能防止关键条目被误关、误删、误改，但还需要进一步人性化。

推荐补充：

- 显示锁定原因
- 解释“锁住后保护了什么”
- 提供“一键复制为可编辑副本”

推荐交互：

- 原条目继续锁定
- 用户点击复制后生成自己的副本
- 副本默认不再继承强锁，或只保留必要保护

这样可以解决：

- 不敢改官方预设
- 想试验又怕改坏
- 只看到禁止会本能排斥的问题

---

## 6. 作用域要讲人话，不要只显示机器值

当前 `scope` 是底层有价值的编译元数据，但对普通作者而言过于抽象。

推荐前台表达：

- 适用于所有会话
- 适用于单人会话
- 适用于多人会话
- 当前会话中：生效 / 未生效

也就是说，前台需要同时显示：

- 规则定义的适用范围
- 当前上下文中的命中状态

这能避免典型困惑：

- 我明明开了，为什么没效果？

---

## 7. 来源解释要轻量，不要做成重工程面板

Night Voyage 是结构化编译器，因此必然存在：

- 预设
- 角色卡
- 状态覆盖层
- 世界书
- 历史总结

之间的权重和覆盖关系。

普通作者并不需要完整 debug 日志，但需要知道“当前这条效果主要来自谁”。

推荐轻量表达：

- 这条风格主要来自预设
- 这条设定主要来自角色卡
- 这条世界设定来自命中的世界书
- 当前状态覆盖了角色初始设定

推荐交互：

- 标签显示静态来源
- hover 气泡显示当前主导来源

---

## 8. 编译预览应尽量图形化、树形化

编译预览不应只是一大段 `systemText`。

更好的形式是：

### 作者视图

- 当前会生效什么
- 哪些规则进入了 system
- 哪些没生效
- 哪些被裁剪了
- 最终效果摘要

### 结构视图

- `system blocks`
- `example messages`
- `params`

长期方向：

**预设编辑与编译预览都应朝树形图形化结构演进。**

也就是说：

- 可以在一个项中添加子项
- 也可以不添加子项
- 平铺 block 编辑可逐步退为高级补充入口

---

## 9. token 成本要做成多来源彩条

后期推荐引入“多来源 token 占比条”，类似 GitHub 的语言占比条。

推荐来源分段：

- 预设
- 角色卡
- 世界书
- 剧情总结
- 最近历史
- 当前输入

推荐效果：

- 不同来源用不同颜色
- 一眼看出谁占得最多
- 一眼看出谁在挤压空间
- 为剧情总结系统和历史裁剪系统提供用户可见反馈

一句话说：

**要让复杂度可见，而不是等效果变差之后才追。**

---

## 10. provider 差异做成轻量提示而不是重面板

Night Voyage 已经有 `provider override` 和能力矩阵方向，但前台不适合直接暴露工程术语。

更合适的交互：

- 通用：无提示
- 有兼容提醒：黄色感叹号
- 明确不支持：红色提示

hover 说明示例：

- 当前 provider 不支持 prefill
- 当前 provider 会忽略某些参数
- 当前 provider 需要兼容模式

推荐补充兼容性徽章：

- 通用
- 依赖 prefill
- 依赖 regex validator
- 依赖 response mode

---

## 11. 预设列表必须从“名字列表”升级为“资产列表”

未来预设一定会越来越多。

因此列表页不能只显示：

- 名字

推荐至少显示：

- 名字
- 一句话效果摘要
- 适用场景
- 内容分级
- 是否推荐
- 是否实验性
- 最近使用时间
- 标签

对于绑定关系，当前不建议强行在预设页做过多反向展示；优先由会话视角来展示“当前绑定了哪个预设 / 世界书”。

---

## 12. 平台应提供“安全试验区”

推荐优先实现：

- 一键复制预设副本

后续可以再扩展：

- 草稿态
- 沙盒保存
- 分支版

但第一步最值得做的是：

**复制当前预设为实验版。**

这能最低成本解决：

- 想试又怕改坏
- 不敢碰正式预设
- 锁定条目带来的心理阻力

---

## 当前建议优先级

### 第一梯队

1. `单人 / 多人`、`SFW / NSFW` 一级互斥主分组
2. 子项气泡说明 + 短例子 + 适合场景
3. 一键复制预设副本
4. provider 黄色感叹号

### 第二梯队

5. 作用域的人话化展示
6. 来源解释标签 + hover 来源说明
7. 树形图形化编译预览

### 第三梯队

8. token 多来源彩条
9. 安全区 / 自由区视觉分区
10. 作者可填写的“注意事项 / 副作用 / 适用说明”

---

## 非目标

当前不推荐：

- 把 Night Voyage 预设重新做回 ST 式大文本拼装工作台
- 用编程思维要求作者理解宏、前后拼块、注入顺序后才会使用
- 为了防错而对作者施加过多硬限制，压缩预设系统发挥空间
- 把 provider 兼容性做成重工程配置面板

---

## 最终一句话总结

**Night Voyage 的预设系统应当让作者像写导演要求那样写预设：明确意图、说明场景、选择效果、理解限制，而不是像写程序那样去思考拼接、宏和内部编译细节。**
